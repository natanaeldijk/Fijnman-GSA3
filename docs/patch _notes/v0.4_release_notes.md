# 📝 Release Notes — v0.4.0

## CFM Modularization & Execution Layer Introduction

---

## 🚀 Overview

This release introduces a modular architecture for the CFM layer, separates the toy core from the formal core, and adds an initial execution layer (VCE). The system is now structured around clearly defined responsibilities between transition logic, diagnostics, and execution.

---

## ✨ Added

### CFM Modules
- src/cfm/api.py
    - Introduces CFMResult as structured output
    - Adds cfm_step as the primary integration point
- src/cfm/diagnostics.py
    - is_rupture
    - interference
    - classify_interference
    - update_pi


### Toy Core
- src/gsa/toy_core.py
    - Minimal deterministic core for testing and experimentation
    - Clean (result, x_new) interface


### VCE (Execution Layer)
- src/vce/
    - Initial minimal runner (minimal.py)
    - Foundation for executable flows

---

## 🔄 Changed

### CFM Architecture
- Refactored proto.py into:
    - diagnostics layer
    - API layer
- Reduced proto.py to compatibility/re-export layer

### Public API
- cfm.__init__ now cleanly exposes:
    - CFMResult
    - cfm_step
    - diagnostic functions

### Core Separation
- gsa.__init__ now distinguishes:
    - GSA51Core (formal core)
    - Core (toy core)

### Tests
- Updated to:
    - use toy_core.Core
    - align with actual CFM behavior
    - use pytest.approx for numeric stability

### Pytest Configuration
- Fixed test discovery:
    - testpaths = tests
    - python_files = test_*.py

---

## 🧹 Removed / Cleaned
- Removed monolithic CFM implementation (~600+ lines) from proto.py
- Eliminated duplicate core definitions in tests/
- Resolved inconsistent core/CFM interfaces

---

## 🧠 Design Improvements

- Clear separation of concerns:
    - Core → state transitions
    - CFM diagnostics → interpretation only
    - CFM API → structured output
    - VCE → execution layer

- Enforced principle:
    - CFM does not modify the core

Standardized interface:
result, x_new = core.step(x, dx)
